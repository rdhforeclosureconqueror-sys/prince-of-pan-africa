# Simba Mutual Aid Pilot Launch Plan

**Status:** Pilot readiness plan for a future approved phase; Phase 1 implements documentation only.

## Pilot Purpose

The pilot should test whether governed community support can operate with dignity, privacy, clear controls, and reliable records after activation.

## Pilot Scope

The pilot should be limited, allowlisted, manually reviewed, manually controlled, and reported in aggregate. It should not include automatic approvals or automated payouts.

## Participants

Participants may include allowlisted members, Mutual Aid Committee reviewers, finance reviewer, privacy steward, records keeper, governance liaison, and admin support.

## Access Rules

Only authorized participants may access pilot workflows. Sensitive request details must be restricted to reviewers with a need to know.

## Pilot Aid Categories

Pilot categories should be narrow, urgent, and stabilizing: housing stabilization, utility shutoff prevention, emergency food support, transportation crisis support, or another category approved by governance.

## Pilot Limits

Set per-request limits, category limits, monthly fund limits, documentation requirements, reviewer quorum, conflict rules, and pause triggers before launch.

## Pilot Workflow

1. Confirm activation conditions.
2. Confirm allowlist.
3. Collect minimum necessary request information.
4. Perform conflict screening.
5. Complete committee review.
6. Complete finance checklist.
7. Record manual processing status if approved.
8. Send respectful decision language.
9. Include aggregate outcome in impact reporting.

## Manual Processes During Pilot

During the pilot, payments, reimbursements, and support processing should remain manual and outside automated product flows unless a later approved phase changes this.

## Data to Collect

Collect request category, timestamps, status changes, documentation checklist completion, recusal records, decision reason codes, finance checklist completion, manual processing reference, appeal status, and aggregate outcome metrics.

## Success Criteria

Success means no privacy incidents, no uncontrolled fund movement, accurate records, timely review, respectful communication, clear governance reporting, and member trust in the process.

## Stop Conditions

Pause the pilot for privacy incidents, accounting discrepancies, reviewer conflicts without alternates, unclear policy, unsafe behavior, unavailable funds, governance concerns, or system access failures.

## Roles and Responsibilities

- Governance: authorizes pilot and receives reports.
- Committee: reviews requests.
- Finance reviewer: confirms controls.
- Privacy steward: protects sensitive information.
- Records keeper: maintains records.
- Admin: coordinates workflow.

## Pilot Timeline

Draft timeline: two weeks pre-pilot readiness, four to six weeks limited pilot, one week pause for review, then governance decision on next steps.

## Pre-Pilot Checklist

- Activation threshold reached.
- Policy approved.
- Governance process approved.
- Accounting controls approved.
- Privacy rules approved.
- Approval controls approved.
- Reviewers trained.
- Language approved.
- Stop conditions accepted.
- Reporting template ready.

## Go/No-Go Rules

Go only if every activation and pre-pilot checklist item is complete. No-go if any required control is missing, unclear, or untested.

## Post-Pilot Review

Review aggregate outcomes, response times, policy issues, privacy incidents, accounting issues, appeals, member feedback, reviewer workload, and recommended changes.

## Required Activation Guardrails

- Mutual Aid Society is currently **Building Toward Activation**.
- The **$20,000 threshold** must be reached before activation.
- Activation also requires approved policy, governance process, accounting controls, privacy rules, and approval controls.
- There are **no mutual aid distributions before activation**.
- Support is reviewed support, not guaranteed support.
- There is no automatic approval.
- There is no cash-equivalent member balance.
- There is no runtime fund movement in this Phase 1 documentation PR.
- There is no member-facing application flow yet.
- There is no payment, payout, or reimbursement logic in this PR.


## Phased Roadmap

1. **Phase 1 — Documentation package.** Create and organize the six documents only.
2. **Phase 2 — Display-only Mutual Aid page.** Future PR only; adds `/mutual-aid` overview page with safe language, activation threshold, coming-soon status, and no request flow.
3. **Phase 3 — Admin/internal planning scaffold.** Future PR only; adds non-public admin planning placeholders if approved, with no live requests or payouts.
4. **Phase 4 — Contribution ledger planning/display.** Future PR only; tracks fund progress only if approved, with no distributions.
5. **Phase 5 — Request intake pilot.** Future PR only; allowlisted members only, no automatic approval, and no automated payouts.
6. **Phase 6 — Committee review pilot.** Future PR only; adds reviewer workflow, conflicts, and decisions.
7. **Phase 7 — Manual disbursement tracking.** Future PR only; records manual payments after governance approval, with no automated payments.
8. **Phase 8 — Pilot reporting.** Future PR only; adds anonymized impact and governance reporting.
